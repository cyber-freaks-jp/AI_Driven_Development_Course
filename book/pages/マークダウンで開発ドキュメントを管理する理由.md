# マークダウンで開発ドキュメントを管理する理由

## なぜExcelやWordではダメなのか？

従来の開発現場では、要件定義書や設計書をExcelやWordで作成するのが一般的でした。しかし、AI駆動開発の時代において、これらのフォーマットには大きな問題があります。

### ExcelやWordの問題点

**1. AIが認識しづらい**

- バイナリ形式（.xlsx、.docx）は人間向けのフォーマット
- AIがテキストを正確に抽出できない
- 表組みやレイアウト情報が邪魔をする
- フォントやスタイル情報が余計なノイズになる

```
Excel形式の例：
[複雑なバイナリデータ] + [表の構造情報] + [フォント情報] + [実際のテキスト]
↓
AIには読み取りにくい
```

**2. バージョン管理が困難**

- Gitで差分を確認できない
- 変更履歴が追跡しづらい
- 競合解決が不可能に近い

**3. 検索性が低い**

- ファイル内の情報をAIが横断的に検索できない
- コンテキスト提供に時間がかかる

## マークダウンの利点

マークダウン形式で開発ドキュメントを管理することで、以下のメリットが得られます。

### 1. AIが100%理解できる

マークダウンは純粋なテキストファイルです。AIは以下のように完璧に理解できます：

```markdown
# 要件定義書

## 機能要件

### ユーザー登録機能
- メールアドレスとパスワードで登録
- メール認証必須
- パスワードは8文字以上
```

↓ AIには明確に伝わる

- 見出し構造が明確
- 箇条書きがわかりやすい
- 余計な情報がない

### 2. Gitで完璧にバージョン管理できる

```bash
# 差分が一目瞭然
- パスワードは6文字以上
+ パスワードは8文字以上
```

- 誰がいつ何を変更したか追跡可能
- ブランチを切って並行作業可能
- レビューしやすい

### 3. 全フェーズでAIに情報を渡せる

マークダウンで統一すれば、開発の全フェーズでAIに正確に情報を伝えられます：

```
要件定義書.md
    ↓ AIに渡す
設計書.md
    ↓ AIに渡す
実装コード
    ↓ AIに渡す
テストコード
```

すべてのフェーズで情報が断絶せず、スムーズに連携できます。

### 4. 図もコードで管理できる（Mermaid）

後述しますが、Mermaidを使えば図もマークダウン内にテキストで記述できます：

```markdown
\`\`\`mermaid
graph TD
    A[ユーザー] --> B[ログイン画面]
    B --> C[ダッシュボード]
\`\`\`
```

↓ 自動でフローチャートが生成される

## 実際の効果

### 従来の方法（ExcelやWord）

```
要件定義書.xlsx → AIに読ませる → エラー多発 → 手動で修正
設計書.docx → AIに読ませる → 表がうまく認識されない → 手動で整理
```

**結果：AIの恩恵を十分に受けられない**

### マークダウン方式

```
要件定義書.md → AIに読ませる → 完璧に理解 → 設計書.mdを自動生成
設計書.md → AIに読ませる → 完璧に理解 → コードを自動生成
```

**結果：開発の全フェーズでAIが最大限活躍**

## 移行は簡単

既存のExcelやWordドキュメントがあっても、移行は簡単です：

### 手動移行

1. ExcelやWordの内容をコピー
2. マークダウンの見出しと箇条書きで整理
3. Gitにコミット

### AIで自動移行

Cursorを使えば、既存のドキュメントをマークダウンに変換することも可能です：

```
プロンプト例：
「この要件定義書の内容をマークダウン形式に変換してください。
見出しレベルを適切に設定し、箇条書きを活用してください。」
```

## まとめ

| 比較項目 | Excel/Word | マークダウン |
|---------|-----------|-------------|
| AI認識性 | △ 低い | ◎ 完璧 |
| バージョン管理 | × 困難 | ◎ 完璧 |
| 検索性 | △ 低い | ◎ 高い |
| 編集の手軽さ | ○ 簡単 | ◎ 非常に簡単 |
| 図の管理 | ○ 可能 | ◎ コードで管理 |

**AI駆動開発では、マークダウン形式が圧倒的に有利です。**
